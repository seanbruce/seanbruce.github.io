JavaScript函数中的this指的到底是什么？

2019年 一月 30

如果你不清楚如何判断 JS 函数中 this 的值是什么，也许你可以花些时间阅读一下这一篇博客。

在解释this值判断问题前，需要简单提一下另外一个同样重要，同样让 JS 新手困惑的知识点(闭包)，因为对闭包的判断常常和对 this值的判断弄混淆。

闭包的判断和this判断最大的不同就是，闭包是在代码编写时就确定的，而 this 值的判定则是当函数调用的时候1才能确定，这一点非常重要, 举个例子：

const names = ['jim', 'Bob', 'Amy']
const funcs = []
function funcGenerator(name)
  return function generatedFunc() {
    console.log("I'm " + name)
  }
}

for (name of names) {
  funcs.push(funcGenerator(name))
}

for (func of funcs) {
  func()
}
// 输出结果
// I'm jim
// I'm Bob
// I'm Amy

第 9 行的循环里，funcGenerator被调用了三次，分别传入jim Bob Amy。返回的新函数存入funcs数组内。注意到到循环结束，生成的函数并没有被调用，而是被保存到了函数定义所处作用域(funcGenerator 内部)的外面。当这三个函数在第 14 行被调用的时候，已经和funcGenerator内的name参数不在同一个作用域中，但是返回的每个函数内部的name引用，任然可以访问name参数的值，这就是闭包在发挥作用，目前，关于闭包你只需要记住上面黑体字的部分。

var name = 'tom'

function foo() {
  var name = 'jim'
  return this.name
}

foo.name = 'ellen'

let bar = {
  name: 'Bob',
  whoAmI() {
    return this.name
  },
}

let baz = bar.whoAmI

foo()
bar.whoAmI()
baz()

在 JavaSript 中函数的this值的判断遵循四条规则，它们之间有优先级的区别，最先被符合条件的规则决定了 this 的值，我按照优先级由低到高的顺序来介绍。

上面代码中第 19 行输出的结果是多少？如果你回答是'ellen'，恭喜你答错了。假设 this 其实指向 function 自身，是常见的误解之一，第 14 行对foo的name属性的赋值并不会影响到 this 的值。

如果你说：“答案是 jim，foo 函数内就定义了 name 这个变量，然后最后返回这个值。“，恭喜你又错了，this的值也不是指向函数的作用域链(尽管有时候，结果看似是一致的，后面会解释)

正确的回答是tom，为什么？

第一条规则

当一个函数被直接的调用时，this 的值指向 window 这个对象，当使用”use strick”模式时，this 的值是 undefined。

这里说的直接调用，指的就是foo()这种形式，第一行代码在全局作用域window下定义了一个 name 的变量，即window.name = 'tom'。this 指向的是window。（这里如果用 ES6 中的 let 关键字声明变量的话，结果会不一样，我推荐你试试这种方法，并找出导致结果不一样的原因）

现在可以解释，为什么输出的结果看起来像是对原型链的访问。有一种假设：如果第 4 行没有在foo内定义一个变量name，那么顺着作用域链去查找的话，就会在 window 这个全局作用域上找到name。 但事实是假设是错误的，结果的一致只是凑巧而已，this的值绝对不是函数的作用域链。

第二条规则，比第一条优先：

当一个函数以对象方法的形式被调用的时候，this 指向这个对象，可以使用 this.加上对象的属性名引用该对象的所有属性。 也可以说是 this 值的隐性绑定(implicit bind)

那么你可以想到，因为上面的规则，第 20 行代码输出的结果是Bob。对象的方法这种说法，来自于面向对象的编程思想。我更倾向不使用”对象的方法”这种说法，因为在 JavaScript 中，一个函数并不被一个对象所拥有，对象只是拥有一个函数的引用，一切都是对象的属性。由此引出一个陷阱。第 17 行代码，把对象的方法保存到对象外一个变量中，结果输出是window中的name，而不是bar中的name。正如前面所说的，对像只是拥有函数的引用，而不是函数本身。函数在何处定义同样不重要，无论是在对象内部被定义后被外部变量引用，还是将在别处被定义的函数的引用保存到对象的属性上，最终决定 this 的，还是函数的调用栈。object.function()只有这种形式的函数调用，this 才指向对象。

上面的代码的最后三行的输出是

  • tom
  • Bob
  • tom
第三条规则，比第二条优先：

从 ES5 开始每个函数的原型上多了两个方法，call和apply，两个方法的作用相同。 用法上，将需要绑定到 this 值的对象做为第一个参数传入(explicit bind)。对于 call 方法， 第一个参数后可以接任意个数的值，这些值会被传入被调用的函数做为参数。而 apply 的第二个参数则是接受一个数组，这个数组会做为参数传入被调用的函数。

function foo() {
  return this.name
}

var bar = {
  name: 'Jobs',
}

foo.call(bar) //输出Jobs
foo.apply(bar) //输出Jobs

var baz = {
  name: 'Jim',
  whoAmI: foo,
}

baz.whoAMI.call(bar) //输出jobs

第 9，10 两行函数调用都使用了显式的 this 绑定。即使没有使用object.function()这种形式调用，函数的 this 值依然指向bar对象。第 17 行函数同时使用了对象的方法和 call 方法，因为 call 的优先级比较高，所以 this 的值是对象 bar，输出结果是jobs。如果多次函数调用需要绑定同样对象，函数后面加上 call 或 apply 这种写法略显繁琐，你可以用一个嵌套函数去返回一个用 apply 或 call 绑定过的函数调用，例如：

function foo() {
  return this.name
}

var obj = {
  name: 'Jobs',
}

function fooBinded() {
  return foo.call(obj)
}

fooBinded() //输出 Jobs

你可以自己写一个工具方法，抽象出函数包装的逻辑，避免代码了的重复并改善代码的可读性。但是更好的选择是使用函数原型上的 bind 方法。它的作用和函数包装 util 方法一样，但逻辑上更为谨慎，考虑到一些特殊情况下函数的表现。

function foo() {
  return this.name
}

var obj = {
  name: 'Jobs',
}

var fooAfterBind = foo.bind(obj)

fooAfterBind() //输出 Jobs

终于介绍到最后一个规则，是优先级最高的一个。

第四条规则

函数调用时，前面加上 new 关键字，就是一次 new 调用。new 调用是 JavaScript 中用来模仿 OOP2中构造函数的调用的语法。new 调用时，会创建一个新的对象，函数内的 this 就指向这个新对象。

function Person(name) {
  this.name = name
}

var person = new Person('Jobs')

person.name //输出 Jobs

即使使用强绑定var BindedPerson = Person.bind({name: 'sean'})，这时调用new BindedPerson('Jobs')返回的对象的 name 属性的值依然是 Jobs。这就是之前提到过 bind 处理的特殊情况。

总结关于 this 值的知识点，判定函数内 this 值依赖四个规则，从优先级高到低分别是：new 函数调用、强绑定bind弱绑定apply``call、方法调用、普通函数调用。第一个符合条件的规则决定了 this 的值。理解 this 对理解 JavaScript 中的面向对象编程是必要的。可能在以后我会写一篇 Javascript 中的面向对象基础的博客。
